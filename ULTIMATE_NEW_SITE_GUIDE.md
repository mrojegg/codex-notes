# ULTIMATE NEW SITE GUIDE

Цель: по этому одному файлу запустить новый сайт как чистый, стабильный и поддерживаемый продукт.

## 0) Что считаем идеальным результатом
- Сайт работает по HTTPS на основном домене.
- `www` корректно редиректит на основной домен.
- Деплой полностью автоматический (через GitHub Actions).
- Любые правки вносятся безопасно (ветка -> проверка -> merge -> автодеплой).
- Есть короткий smoke-check после каждого релиза.

---

## 1) Старт нового проекта (обязательно)
1. Создать отдельную папку проекта.
2. Положить основной файл как `index.html`.
3. Инициализировать git и `main`.
4. Создать отдельный GitHub-репозиторий под этот сайт.

Команды:
```bash
git init -b main
git add .
git commit -m "Initial setup"
```

---

## 2) Настройка автодеплоя (GitHub Pages)
1. Добавить workflow деплоя (`.github/workflows/deploy-pages.yml`).
2. Запушить `main` в GitHub.
3. В репозитории: `Settings -> Pages -> Source: GitHub Actions`.
4. Вкладка `Actions`: убедиться, что workflow зелёный.

Критично для многостраничного сайта:
- В артефакт деплоя должны попадать **все** HTML-файлы, не только `index.html`.
- Без этого вторичные страницы (`/services.html`, `/about.html`) будут давать `404`.

Рекомендуемый шаг в workflow:
```bash
mkdir -p _site
find . -maxdepth 1 -name '*.html' -exec cp {} _site/ \;
if [ -d assets ]; then cp -R assets _site/assets; fi
if [ -f CNAME ]; then cp CNAME _site/CNAME; fi
```

Если видите ошибку `Get Pages site failed ... Not Found`:
- Pages ещё не активирован, зайдите в `Settings -> Pages`, сохраните источник ещё раз и rerun jobs.

## 4) CAA/SSL (если есть проблемы с сертификатом)
Добавить CAA на корень (`@`):
- `CAA 0 issue "letsencrypt.org"`
- `CAA 0 issue "digicert.com"`
- если провайдер сертификата требует: `CAA 0 issue "globalsign.com"`

Примечание: TXT записи верификации (например `_globalsign-domain-verification=...`) обычно допустимы и не мешают.

## 5) Безопасный процесс правок (жёсткое правило)
Любая правка только так:
1. Создать ветку: `feature/<name>`.
2. Внести изменения.
3. Локально проверить сайт.
4. Commit в ветке.
5. Merge в `main` только после проверки.
6. Публикация в прод — только из `main`.

Базовые команды:
```bash
git checkout -b feature/some-change
# edits
git add .
git commit -m "Update"
git checkout main
git merge feature/some-change
git push
```

Если нужно экстренно править `main` (по запросу владельца):
- фикс только точечный,
- сразу после фикса прогнать smoke-check,
- отдельной записью зафиксировать причину обхода стандартного flow.

---

## 6) Обязательный smoke-check после каждого релиза
Проверить вручную:
1. `https://<domain>` открывается.
2. `https://www.<domain>` редиректит на основной.
3. HTTPS активен.
4. Работают меню/модалки/кликабельные элементы.
5. Работают email-ссылки (`mailto` + fallback, если предусмотрено).
6. На мобильных всё кликабельно и читаемо.
7. Все вторичные страницы открываются напрямую (`/services.html` и т.д.) без `404`.

---

## 7) Частые ошибки и быстрые фиксы
1. `DNS check unsuccessful`:
- DNS ещё не применился или зона/NS неверные.
- Проверить NS и A/CNAME, подождать 15-60 мин (иногда до 24-48 ч), нажать `Check again`.

2. `404 There isn't a GitHub Pages site here`:
- Pages не включен или деплой не прошёл.
- Включить Pages source на GitHub Actions и rerun workflow.

3. `404` только на вторичной странице (`/services.html`), а главная работает:
- В workflow в `_site` копируется только `index.html`.
- Исправить шаг публикации: копировать все `*.html`.

4. Push ошибка `workflow scope`:
- PAT без `workflow` scope не может пушить workflow-файлы.
- Нужен токен с `repo` + `workflow`.

5. Push ошибка `Invalid username or token`:
- Токен неверный/истёк.
- Выпустить новый PAT.

6. На iPhone сайт открыт как файл (`file://`) и часть действий не работает:
- Открывать сайт только через `http(s)`.

7. В выпадающем меню текст "разъезжается" и накладывается:
- Причина: слишком общий CSS-селектор (`nav ul`) задевает внутренние списки.
- Фикс: ограничить селектор шапки до `nav > ul` и `nav > ul > li`.

8. На iPhone Safari в выпадающем блоке не виден конец текста:
- Добавить для меню на mobile `max-height` + внутренний скролл:
  - `max-height: 72vh; overflow-y: auto; -webkit-overflow-scrolling: touch;`

9. Hover-меню (`О нас`, `Контакты`) не открывается на телефоне:
- На touch нет hover.
- Нужен mobile-режим по `click/tap` с закрытием по тапу вне блока.

---

## 8) Безопасность (обязательно)
1. Никогда не хранить PAT в `git remote`.
2. Любой токен, попавший в чат/логи, сразу считать скомпрометированным и отзывать.
3. Использовать минимальные scope токена.
4. Для изменений workflow заранее готовить PAT с `repo` + `workflow`.
5. Держать в репозитории только то, что нужно прод-сайту.

Команда очистки remote после push с токеном:
```bash
git remote set-url origin https://github.com/<user>/<repo>.git
```

---

## 9) Что хранить в repo сайта, а что в repo знаний
В repo сайта хранить только прод-артефакты:
- `index.html`
- дополнительные страницы (`services.html` и т.п.)
- `assets/`
- `CNAME` (если нужен)
- `.github/workflows/deploy-pages.yml`
- минимум технических файлов

В repo знаний (`codex-notes`) хранить:
- чеклисты,
- playbook,
- postmortem/ошибки,
- процессы и шаблоны.

---

## 10) Definition of Done (проект завершён)
Проект завершён, когда:
1. домен и HTTPS работают,
2. деплой автоматический и стабилен,
3. smoke-check проходит,
4. процесс правок безопасный и повторяемый,
5. документация для восстановления контекста существует.

---

## 11) Зафиксированные рабочие договорённости
1. Этот файл (`ULTIMATE_NEW_SITE_GUIDE.md`) — единственный источник знаний в репозитории `codex-notes`.
2. Новые материалы (ошибки, фиксы, правила, улучшения процесса) добавляются только в этот файл.
3. Новые отдельные обучающие файлы в `codex-notes` не создаём.
4. Фокус работы: только рабочие процессы и написание кода.
5. Для каждого нового сайта используем этот же процесс: отдельный проект, отдельный репозиторий, безопасные правки через ветки, автодеплой из `main`.
6. Репозиторий сайта хранит только прод-файлы, методология и опыт — только здесь.
7. Если нужно не трогать локальную рабочую папку, фиксы делаются в отдельном чистом clone и пушатся после проверки.

## 13) Критичные уроки из проекта «Кубикор» (март 2026)
1. Нельзя ограничивать Pages-артефакт одним `index.html`, если сайт становится многостраничным.
2. Любой hover-only интерфейс нужно дублировать tap-логикой для iPhone/Android.
3. Выпадающие панели с большим текстом обязаны иметь внутренний mobile-scroll.
4. CSS шапки должен быть изолирован от контента (никаких широких `nav ul` для всего дерева).
5. Перед релизом проверять не только главную, но и прямые URL всех внутренних страниц.
6. Любой показанный токен сразу отзывается и перевыпускается.
7. Правки workflow требуют отдельного токена с `workflow` scope, иначе пуш будет отклонён.
